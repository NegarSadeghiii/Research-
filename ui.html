2. Mathematical Formulation

MILP
Minimize total supply chain cost subject to routing, capacity, and deadline constraints.
Objective
min Σp Costfacility
  + Σp Costtransport
  + Σp Costmaterial+QC
  + M × Σp Lateness[p]
Key Constraints
Each patient: exactly 1 route
Patient ci returns to hi
Open facilities ≤ MAX_FAC
Patients in mfg at time t ≤ FCAP[m]
LATE[p] ≥ TRT[p] - DEADLINE[p]

DEADLINE[p] = base_duegroup + tolerancegroup × (1 - τ)

5. Solution Evolution

3,500x FASTER
Step 1: Original Model (Negar Sadeghi)
Pyomo AbstractModel with soft due-date penalties. Decision variables Y1[p,c,m,j,t] and Y2[p,m,h,j,t] track patient flows at every time period. Solved with Gurobi (commercial license required). The model was extended from the i-SHIPMENT formulation with patient-specific due dates.
Step 2: What We Tried (in order)
We tried four different approaches, one at a time, each building on lessons from the previous failure. All four solve the same underlying problem (same patients, same constraints, same objective). The only difference is how the solver represents and searches for the solution. Think of it like four different ways to search a phone book — the answer is the same, but some methods are faster.
Attempt 1: Original Pyomo Model (Negar's)
Creates a binary variable for every possible (patient, facility, transport, day) combination. With 50 patients and 130 days, that's 288,000 binary variables. The solver must explore an astronomical number of combinations. Result: timeout even after 5 minutes. Needs Gurobi (commercial, $$$).
Attempt 2: Time-Windowed MIP
Same Pyomo approach, but we noticed most variables are guaranteed to be zero (a patient arriving day 50 doesn't need variables for days 1-49). So we only create variables within each patient's feasible time window. 10x faster, solves 15 patients, but still times out at 50. Same math, fewer wasted variables.
Attempt 3: Two-Phase Decomposition
Split the problem: first relax all binary variables to continuous (LP) to quickly identify the best 2 facilities. Then fix those facilities and solve only the patient routing. First approach to solve 50 patients (351 seconds), but the solution isn't proven optimal — we picked facilities based on an approximation.
Attempt 4: CP-SAT Route Reformulation (final)
Completely different approach. Instead of tracking flows across time periods, we ask a simpler question: "which route does each patient take?" Each patient picks one of 24 routes (6 facilities x 2 outbound x 2 return transport). Capacity is enforced using interval scheduling. 50 patients in 0.1 seconds, proven optimal.
Does this change the problem? No. All four approaches solve the same optimization: minimize cost, respect capacity, meet deadlines. The CP-SAT model makes one realistic assumption: patients' cells are shipped immediately after collection (no holding). This is medically correct — harvested cells degrade. But we also added a "wait" variable (0-10 days) so the solver CAN delay shipment when needed to balance facility load.
Step 3: Why CP-SAT is So Fast
Imagine you're planning seating for a wedding dinner. The old approach (Pyomo) is like assigning each guest to a specific chair at a specific time — "Guest A sits in Chair 7 at Table 3 at 7:15 PM." That's millions of possible combinations.

CP-SAT's approach is: "Guest A goes to Table 3." Period. Which chair and when they sit down is automatically determined by the table they're assigned to. This is 240 times fewer decisions to make.

Technically, here's what happens:
1.
Each patient has 24 possible routes (6 facilities x 2 outbound transport x 2 return transport). The solver picks exactly one.
2.
Once a route is picked, the turnaround time is known (e.g., j1+m1+j1 = 17 days). No need to track individual days.
3.
Capacity is checked using interval scheduling: "if Patient A and Patient B both use facility m1 and their manufacturing windows overlap, count them." The solver ensures no facility exceeds its capacity at any point in time.
4.
CP-SAT uses constraint propagation: if Patient A must go to m1 (because m1 is the only open facility close enough), it immediately eliminates all other options for Patient A. This cascades, pruning the search space massively.
5.
The solver runs on all CPU cores in parallel, each trying a different search strategy simultaneously.
